IIS 資料夾權限的設定方式

  1. 常見的情境 1 Windows 驗證
  2. 常見的情境 2 匿名驗證
  3. 結論

IIS (Internet Information Server) 的權限設定只是幾個小按鈕就能完成，但正確的設定就能避免浪費除錯時間。而在設定時最基本的情境就是要讓網站能夠如期呈現，並且讓對應權限的使用者進行符合權限的動作。

常見的情境 1 Windows 驗證

開發應用系統，使用 Windows 驗證，應用程式會利用 Identity.User 的資訊與資料庫中的權限管理做系統權限的驗證。同時伺服器應用程式所在的資料夾必須支援共用資料夾，如此便可以藉由 UNC 的方式發布編譯後的程式至伺服器。

權限設定

NTFS 安全性

  • Domain\Authenticated Users(Read / Write)
  • Machine\IIS_IUSRS (Read) 或 AppPool\PoolName (Read)

資料夾共用

  • 進階共用移除 Everyone
  • 進階共用加入 Domin\AD groups (Read /Wrtie)

完成設定後系統會比對進階共用與 NTFS 安全性權限的交集，因此一般使用者僅能透過 IIS_IUSRS 進行瀏覽，而屬於 AD groups 的開發人員可以透過 UNC 存取發布程式的資料夾。

如果有上傳檔案的需求，則需要另外對上傳的資料夾提供 IIS_IUSRS (Write) 的權限，使用者方可藉由 IIS_IUSRS 將檔案上傳至伺服器。

常見的情境 2 匿名驗證

僅需要匿名驗證時，則加入 IUSR 並提供 Read 的權限。

權限設定

NTFS 安全性

  • Domain\Authenticated Users(Read / Write)
  • Machine\IIS_IUSRS (Read)

結論

如果資料夾層層迭代，同時會使用到 Windows 驗證與匿名驗證，並且有共用資料夾需求時，則可以實踐的設定方式如下：

NTFS 安全性

  • Domain\Authenticated Users(Read / Write)
  • Machine\IIS_IUSRS (Read) 或 AppPool\PoolName (Read)
  • Machine\IUSR (Read)

資料夾共用

  • 進階共用移除 Everyone
  • 進階共用加入 Domin\AD groups(Read / Write)
